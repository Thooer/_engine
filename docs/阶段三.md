### 3. 引擎 main 入口

```rust
// crates/engine-app/src/main.rs

use std::path::PathBuf;
use clap::Parser;

#[derive(Parser, Args)]
struct Args {
    /// 项目路径（相对于引擎根目录）
    #[arg(default_value = "./demos/simple_demo")]
    project: PathBuf,
}

fn main() {
    let args = Args::parse();
    
    // 获取项目绝对路径
    let project_path = if args.project.is_absolute() {
        args.project
    } else {
        std::env::current_dir()
            .unwrap()
            .join(&args.project)
    };
    
    tracing::info!("Running project: {:?}", project_path);
    
    // 创建并运行引擎
    let mut app = ToyEngineApp::new(project_path);
    app.run();
}
```

### 4. 内置 Shader 机制

#### 4.1 内置 Shader 模块

```rust
// crates/engine-renderer/src/graphics/builtin_shaders.rs

/// 内置 shader 列表
pub enum BuiltinShader {
    BasicDiffuse,
    Line,
    Unlit,
}

impl BuiltinShader {
    /// 获取 shader 源码
    pub fn source(&self) -> &'static str {
        match self {
            Self::BasicDiffuse => include_str!("../../assets/builtin/basic_diffuse.wgsl"),
            Self::Line => include_str!("../../assets/builtin/line.wgsl"),
            Self::Unlit => include_str!("../../assets/builtin/unlit.wgsl"),
        }
    }
    
    /// shader 标识符（用于材质配置引用）
    pub fn identifier(&self) -> &'static str {
        match self {
            Self::BasicDiffuse => "builtin/basic_diffuse",
            Self::Line => "builtin/line",
            Self::Unlit => "builtin/unlit",
        }
    }
}
```

#### 4.2 PipelineGenerator 集成

修改 `PipelineGenerator`，在加载文件系统 shader 之前先注册内置 shader：

```rust
impl PipelineGenerator {

    fn register_builtin_shaders(&mut self) {

```

## 七、WASM 脚本系统

### 概述

WASM 脚本系统用于运行时逻辑热更新，允许用户编写 Rust 代码并编译为 WASM，在引擎运行时动态加载和执行。

### 当前状态

- ✅ WASM 运行时框架已实现（`engine-app/src/wasm_runtime.rs`）
- ✅ 支持加载 `.wasm` 文件
- ⚠️ Windows 平台存在内存对齐问题，需要修复
- ✅ 轨道相机控制在 `on_update` 中实现

### 使用方法

1. 创建 WASM 脚本项目（参考 `demos/simple_demo/assets/scripts/`）
2. 编译为 WASM: `cargo build --release --target wasm32-unknown-unknown`
3. 在 `project.toml` 中配置 `script = "path/to/script.wasm"`
4. 引擎启动时自动加载

### 相机模式

在 `project.toml` 中配置 `camera_mode` 选项：

```toml
camera_mode = "orbit"     # 圆形轨道（默认）
camera_mode = "orbital"   # 圆形轨道 + 垂直摆动
camera_mode = "figure8"   # 8字形轨迹
camera_mode = "spiral"    # 螺旋上升
```

### 已实现函数

WASM 脚本可导出以下函数供引擎调用：

- `update(dt, frame, radius, height, speed)` - 圆形轨道
- `update_orbital(...)` - 轨道 + 垂直摆动
- `update_figure8(...)` - 8字形轨迹
- `update_spiral(...)` - 螺旋上升
- `get_camera_x/y/z()` - 获取相机位置

参数和返回值都使用 i32 传输 f32 的位表示。

### 已知问题

- Windows + wasmer 2.x 存在 `ptr::copy` 内存对齐问题 ✅ 已修复（升级到 4.x）

## 九、架构问题深度分析与改造思路

经过对整个引擎代码的深度调研，发现以下**架构层面的抽象错误**及对应的改造方案。

---

### 9.1 Engine 结构体 - 核心抽象错误

#### 问题描述

`Engine` 结构体存在**职责不清**和**数据所有权的混乱**：

```rust
// crates/engine-app/src/lib.rs:30-39
pub struct Engine {
    window: Option<&'static Window>,           // 窗口引用
    ctx: Option<SurfaceContext<'static>>,     // GPU 上下文
    input: InputState,                        // 输入状态
    pub world: World,                         // ECS World
    pub main_renderer: Option<MainRenderer>,  // 渲染器实例
    pub exit_requested: bool,
    pub frame_index: u32,
}
```

**具体问题**:

1. **所有权混乱**: `Engine` 拥有 `World`、`ctx`、`main_renderer`，但 `window` 却是引用
2. **生命周期问题**: `ctx` 和 `window` 的生命周期标记为 `'static`，但实际是 `'a`
3. **双重状态**: `input` 同时存在于 `Engine` 中和 `World` 中（作为 Resource）
4. **trait 实现分裂**: `EngineTrait` 暴露 getter，但 `Engine` 本身是可变的

#### 修复状态 ✅ 已修复

- 已移除 `Engine.input` 字段
- 已移除 `EngineTrait.input()` 方法  
- 现在只使用 ECS Resource 方式访问 InputState：`world.get_resource::<InputState>()`
- 已更新所有调用方（ecs_scene_demo.rs, camera.rs）

---

### 9.2 App Trait - 接口污染

#### 问题描述

`App` trait 承担了过多职责，混合了多个抽象层次的概念：

```rust
// crates/engine-app/src/lib.rs:56-87
pub trait App {
    fn systems(&mut self) -> SystemSchedule;        // ECS 系统注册
    fn configure_schedule(&mut self, schedule: &mut SystemSchedule) {}  // 调度器配置
    fn on_start(&mut self, engine: &mut Engine) {}   // 启动回调
    fn on_window_event(&mut self, engine: &mut Engine, event: &WindowEvent) {}  // 窗口事件
    fn on_resize(&mut self, engine: &mut Engine, new_size: SurfaceSize) {}  // 窗口大小变化
    fn on_update(&mut self, engine: &mut Engine, dt_seconds: f32) {}  // 每帧更新
    fn on_render(&mut self, engine: &mut Engine) {}  // 渲染回调
    fn configure_ecs(&mut self, world: &mut World) {}  // ECS 配置
}
```

**具体问题**:

1. **接口膨胀**: 11 个方法中只有 2-3 个是必须的
2. **生命周期混用**: `engine: &mut Engine` 意味着 App 必须知道 Engine 的内部结构
3. **回调模式 vs 响应式**: 使用回调而非 ECS 系统响应事件
4. **与 Bevy App 不兼容**: 无法迁移到更成熟的 ECS 调度器

#### 改造方案

```rust
// 方案 A: 精简 App trait
pub trait App {
    fn setup(&mut self, world: &mut World);  // 初始化
    // 系统注册移到 SystemSchedule 层
}

// 方案 B: 使用 Plugin 模式
pub trait Plugin {
    fn build(&self, app: &mut ToyEngine);
}
```

### 9.3 SystemSchedule - 过度简化的调度器 ✅ 已增强

#### 问题描述

当前调度器过于简单，缺少现代 ECS 的核心功能：

```rust
// 修复前
pub struct SystemSchedule {
    stages: HashMap<SystemStage, Vec<SystemFn>>,
    // ...
}
```

**具体问题**:

1. **无条件执行**: 没有 `run_if()` 机制
2. **无依赖声明**: 系统执行顺序无法声明依赖关系
3. **无并行执行**: 所有系统串行执行

#### 修复状态 ✅ 已增强

已添加条件系统支持：

```rust
// 修复后 - 添加了 ConditionalSystem
pub struct ConditionalSystem {
    pub system: SystemFn,
    pub condition: Option<SystemCondition>,  // 新增：条件
}

// 新增方法：add_system_with_condition
pub fn add_system_with_condition<F, C>(
    &mut self, system: F, stage: SystemStage, condition: C
) -> &mut Self
where
    F: FnMut(&mut World) + Send + Sync + 'static,
    C: Fn(&World) -> bool + Send + Sync + 'static;

// 使用示例
schedule.add_system_with_condition(
    my_system,
    SystemStage::Update,
    |world| world.get_resource::<SomeFlag>().map(|f| f.enabled).unwrap_or(false)
);
```

### 9.4 渲染管线 - 硬编码的 Pass 顺序

#### 问题描述

渲染 passes 的执行顺序在代码中写死：

```rust
// crates/engine-renderer/src/renderer/RendererTrait_MainRenderer.rs:429-437
fn render<C: SurfaceContextTrait>(&mut self, ctx: &mut C) {
    // 顺序执行，硬编码！
    MeshForwardPass.render(...);
    LinePass.render(...);
    render_ui(...);
}
```

**具体问题**:

1. **无法动态添加 Pass**: 用户无法在不修改引擎代码的情况下添加后处理
2. **无 Pass 间数据传递**: 无法实现延迟渲染（G-Buffer）
3. **无资源复用**: 每个 Pass 可能重复创建相同的 GPU 资源
4. **与 Render Graph 理念背道而驰**: 现代引擎都使用渲染图

#### 改造方案

```rust
// 引入 Render Graph
pub struct RenderGraph {
    nodes: HashMap<String, Box<dyn RenderNode>>,
    edges: Vec<RenderEdge>,  // 数据流
}

pub trait RenderNode {
    fn execute(&self, ctx: &mut RenderContext);
    fn inputs(&self) -> Vec<ResourceSlot>;
    fn outputs(&self) -> Vec<ResourceSlot>;
}
```

### 9.6 输入系统 - 未接入 ECS

#### 问题描述

`InputState` 未作为 Resource 插入 ECS：

```rust
// crates/engine-app/src/lib.rs:33
pub struct Engine {
    // ...
    input: InputState,  // 独立存储
    // ...
}

// 但也插入了 World
world.insert_resource(InputState::new());
```

**具体问题**:

1. **双重存储**: 既在 Engine 中又在 World 中
2. **无法使用 Query**: 无法在 ECS 系统中查询输入
3. **无法响应输入变化**: 没有 `Changed<InputState>` 机制

#### 改造方案

```rust
// 移除 Engine.input，只保留 ECS 方式
impl ToyEngine {
    pub fn update(&mut self, dt: f32) {
        // 将窗口事件转换为 ECS 事件
        for event in &self.window_events {
            self.world.send_event(event);
        }
        // 运行系统
        self.schedule.run(&mut self.world);
    }
}

// 输入查询方式
fn input_system(query: Query<&InputState>) { /* ... */ }
```

### 9.7 资源管理 - 缺失抽象层

#### 问题描述

资源管理分散在多个地方：

1. `MainRenderer` 中的 `model_cache`、`material_cache`、`texture_cache`
2. `AssetServer` 的简单实现
3. 无统一的资源生命周期管理

**具体问题**:

1. **同步加载**: 无异步加载支持
2. **无依赖追踪**: 场景 A 依赖材质 B，无法自动追踪
3. **无热重载**: 修改资源后需重启
4. **缓存策略缺失**: 无 LRU/引用计数等策略

#### 改造方案

```rust
// 引入统一资源系统
pub trait AssetLoader<T: Asset> {
    fn load(&self, bytes: &[u8]) -> Result<T, AssetError>;
    fn dependencies(&self, asset: &T) -> Vec<AssetId>;
}

pub struct AssetServer {
    loaders: HashMap<TypeId, Box<dyn AssetLoader>>,
    assets: Cache<AssetId, Arc<dyn Asset>>,
}
```

### 9.8 模块依赖 - 循环依赖风险

#### 问题描述

当前模块依赖关系存在问题：

```
engine-app → engine-renderer → engine-core
                              ↘ engine-physics → engine-core
                              ↘ engine-events
```

**具体问题**:

1. **engine-renderer 依赖过多**: 依赖 core、events、scene
2. **app 与 core 边界模糊**: `InputState` 应该在哪个 crate？
3. **scene 职责不清晰**: 是独立的子系统还是渲染的附属？

#### 改造方案

```
清晰的分层：
├── engine-core (基础类型、ECS 组件)
├── engine-platform (窗口、输入抽象)
├── engine-physics (物理)
├── engine-renderer (渲染)
├── engine-scene (场景序列化)
└── engine-app (粘合层)
```

## 十一、总结

ToyEngine 当前处于**原型验证阶段**，核心功能已跑通，但在架构层面存在以下核心问题：

1. **抽象层次混乱**: Engine/App/World 职责不清
2. **调度器过于简化**: 缺少现代 ECS 必备功能
3. **子系统集成不完整**: 物理、输入、渲染各有各的"两层皮"
4. **资源管理缺失**: 无统一抽象

建议按照上述优先级逐步改造，优先解决 P0 问题，确保架构的可持续演进。
