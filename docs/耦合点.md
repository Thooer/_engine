# ToyEngine 架构分析：错误抽象、错误耦合与待抽象耦合

## 一、项目结构总览

```
toyengine/
├── Cargo.toml (Workspace 配置)
├── crates/
│   ├── engine-core/      # 核心层：ECS基础组件、输入、相机
│   ├── engine-renderer/  # 渲染层：图形、材质、UI
│   ├── engine-physics/   # 物理层：Rapier封装
│   ├── engine-events/    # 事件系统
│   ├── engine-scene/     # 场景序列化
│   └── engine-app/       # 应用入口
└── demos/
```

**依赖关系**：
- `engine-app` → `engine-core`, `engine-renderer`, `engine-physics`, `engine-scene`
- `engine-renderer` → `engine-core`
- `engine-physics` → `engine-core`
- `engine-scene` → `engine-core`, `engine-renderer`, `engine-physics`

---

## 二、错误抽象 (Wrong Abstractions)

### 1. ECS 组件分散在多个 crate 中（最严重）

**问题描述**：组件定义位置违背了低耦合原则。

| 组件类型 | 所在位置 | 问题 |
|---------|---------|------|
| `Transform` | `engine-core` | 物理引擎需要引用 |
| `Renderable` | `engine-core` | 基础颜色渲染 |
| `MeshRenderable` | `engine-renderer` | 网格渲染 |
| `Camera2D/3D` | `engine-core` | 基础相机数据 |
| `CameraController` | `engine-renderer` | 轨道相机控制**不应在渲染器** |
| `CameraPriority` | `engine-renderer` | 相机优先级**不应在渲染器** |
| `LineRenderable` | `engine-renderer` | 线条渲染**不应在渲染器** |
| `EcsPointLight` | `engine-renderer` | 光源组件 |
| `EcsDirectionalLight` | `engine-renderer` | 光源组件 |
| `RigidBody` | `engine-physics` | 物理组件 |
| `Collider` | `engine-physics` | 物理组件 |
| `PhysicsHandle` | `engine-physics` | 内部句柄 |

**根本原因**：将"属于同一概念实体的组件"人为拆分到不同 crate。

```rust
// crates/engine-renderer/src/ecs/mod.rs
//! ECS Rendering Components
//!
//! 渲染相关的 ECS 组件，用于将渲染数据与 ECS World 集成
```

这导致：
- `engine-physics` 必须依赖 `engine-core` 获取 `Transform`
- `engine-renderer` 必须依赖 `engine-core` 获取 `Camera3D`
- 循环依赖风险：`engine-scene` 同时依赖 `engine-renderer` 和 `engine-physics`

- `engine-scene` 直接调用 `engine_renderer::graphics::ModelLoader`

### 3. Engine 结构体承担了过多职责

**文件位置**：`crates/engine-app/src/lib.rs`

```rust
pub struct Engine {
    window: Option<&'static Window>,
    ctx: Option<SurfaceContext<'static>>,
    // input 已移除，只使用 ECS Resource: world.get_resource::<InputState>()
    pub world: World,
    /// 主渲染器实例 (由 RenderPlugin 或 App 初始化)
    pub main_renderer: Option<MainRenderer>,
    pub exit_requested: bool,
    pub frame_index: u32,
}
```

**问题**：
- `Engine` 既是"世界容器"(`World`)
- 又是"窗口管理"(`window`, `ctx`)
- 又是"渲染器宿主"(`main_renderer`)
- 职责严重不单一

## 三、错误耦合 (Wrong Couplings)

### 1. engine-scene 双向耦合渲染和物理

**文件**：`crates/engine-scene/src/lib.rs`

```rust
use engine_renderer::graphics::ModelLoaderTrait;
```

和

```rust
fn collect_physics_data(world: &World, entity: Entity) -> Option<PhysicsData> {
    let rigid_body = world.get::<engine_physics::RigidBody>(entity)?;
    let collider = world.get::<engine_physics::Collider>(entity)?;
```

**问题**：
- `engine-scene` 是一个"数据格式" crate，但它直接依赖了 `engine-renderer`（图形实现）和 `engine-physics`（物理实现）
- 这意味着：修改渲染器实现可能会影响场景序列化

---

### 2. 渲染器直接查询 engine-core 的组件

**文件**：`crates/engine-renderer/src/renderer/mod.rs`

```rust
use engine_core::ecs::{Camera3D, Transform};
```

这导致：
- 渲染器知道 ECS 的存在
- 每次 `engine-core` 添加组件，`engine-renderer` 可能需要更新

---

### 3. 主程序直接操作 ECS 实体

**文件**：`crates/engine-app/src/main.rs`

```rust
// 1. 清空 ECS 世界中的实体
{
    let mut world = engine.world_mut();
    let entities: Vec<bevy_ecs::entity::Entity> = world.query::<bevy_ecs::entity::Entity>()
        .iter(&world)
        .collect();
    let count = entities.len();
    for entity in &entities {
        world.despawn(*entity);
    }
```

和

```rust
// 更新相机位置
let mut query = engine.world.query::<&mut engine_core::ecs::Camera3D>();
if let Some(mut camera) = query.iter_mut(&mut engine.world).next() {
    camera.position = pos;
}
```

**问题**：
- 主程序 `main.rs` 不应该直接操作 ECS 内部
- 这破坏了"引擎与游戏逻辑分离"的原则

---

### 4. 物理系统硬编码输入依赖

**文件**：`crates/engine-physics/src/lib.rs`

```rust
use winit::keyboard::KeyCode;

/// 标记实体在按键触发时重置
#[derive(Debug, Clone, Copy, Component)]
pub struct ResetOnKeyPress {
    pub key: KeyCode,
    pub reset_position: glam::Vec3,

```

**问题**：
- 物理 crate 依赖了 `winit`（平台层窗口库）
- 这是反向依赖：物理不应该知道窗口的存在

---

## 四、待抽象的耦合 (Missing Abstractions)

### 1. 缺少"渲染上下文"抽象

**当前**：`MainRenderer` 直接持有 `wgpu::Device` 和 `wgpu::Queue`

```rust
// 1. 创建渲染器
let mut renderer = MainRenderer::new(ctx, engine.window());
```

**应抽象为**：
- `trait GraphicsContext`：抽象 GPU 设备访问
- `trait SceneLoader`：抽象场景加载（当前 `engine-scene` 充当了多个角色）

---

### 2. 缺少"输入系统"抽象

**当前**：`engine-core` 直接依赖 `winit`

```rust
/// 输入系统 v0（键盘为主）
pub mod input;
```

`engine-physics` 也直接使用 `winit::keyboard::KeyCode`

**应抽象为**：
- `trait InputSystem`：抽象输入设备
- `InputCode` 枚举：平台无关的输入码

---

### 3. 缺少"插件注册表"抽象

**当前**：插件在 `engine-app` 中硬编码

```rust
impl App for ToyEngineApp {
    fn systems(&mut self) -> SystemSchedule {
        let mut schedule = SystemSchedule::new();
        physics_plugin().build(&mut schedule);
        render_plugin()
            .with_materials(...)
            .with_scene(...)
            .build(&mut schedule);
        schedule
    }
}
```

**应抽象为**：
- `trait EnginePlugin`：引擎插件接口
- `Engine::register_plugin<T: EnginePlugin>()`

---

### 4. 缺少"配置系统"抽象

**当前**：在 `main.rs` 中手动解析 TOML

```rust
impl ProjectConfig {
    /// 从 project.toml 加载
    fn load_from_path(path: &PathBuf) -> Self {
        // 手动解析 toml::Value
```

**应抽象为**：
- `trait ConfigLoader<T: Config>`
- `EngineConfig`、`ProjectConfig` 实现此 trait

---

### 5. 缺少"热重载"抽象

**当前**：`main.rs` 中手动实现热重载逻辑

```rust
/// 重新加载项目（热重载）
fn reload_project(&mut self, new_project_path: PathBuf, engine: &mut Engine) {
    // 1. 清空 ECS 世界
    // 2. 重新加载配置
    // 3. 重新加载 WASM
    // 4. 重新加载场景和材质
    // ...
}
```

**应抽象为**：
- `trait HotReloader`
- `Engine::reload_project()`

---

### 6. engine-events 可能冗余

**观察**：
- `engine-events` 定义了自己的事件系统
- 但实际上项目主要使用 Bevy ECS 内置的 `Event` 系统

```rust
// 使用 Bevy 的 Event trait 作为基础
pub use bevy_ecs::event::Event;
```

**建议**：评估是否需要独立的事件 crate，或者将其合并到 `engine-core`

---

## 五、总结与重构建议

### 架构改进优先级

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P0** | ECS 组件分散 | 考虑创建 `engine-ecs` 或将组件统一到 `engine-core` |
| **P0** | engine-scene 过度耦合 | 拆分 `SceneLoader` trait，将 GPU 资源加载移至渲染器 |
| **P1** | Engine 职责过多 | 拆分 `Engine` 为 `EngineHost`、`RenderContext`、`PhysicsContext` |
| **P1** | 插件系统位置错误 | 迁移到独立的 `engine-plugin` crate |
| **P2** | 输入系统依赖 winit | 抽象输入层 |
| **P2** | 配置系统手写解析 | 使用 `serde` + 专用 crate |
| **P3** | 评估 engine-events | 合并到 `engine-core` 或移除 |

### 推荐的目标架构

```
toyengine/
├── crates/
│   ├── engine-core/        # ECS框架、基础组件、资源
│   ├── engine-platform/   # winit 封装（输入、窗口）
│   ├── engine-renderer/   # 仅图形渲染
│   ├── engine-physics/    # 仅物理模拟
│   ├── engine-scene/      # 仅场景数据格式
│   ├── engine-plugin/     # 插件系统
│   └── engine-app/        # 应用入口（极简）
```
