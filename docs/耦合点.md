# ToyEngine 深度分析报告

## 一、项目概览

这是一个 **Rust 编写的模块化游戏引擎**，采用 Cargo Workspace 架构，包含 7 个核心 crate 和 2 个演示项目。

### 1.1 整体架构

```
ToyEngine/
├── crates/                          # 引擎核心 (7个crate)
│   ├── engine-core/                  # ECS、资源、输入、相机、插件系统
│   ├── engine-renderer/              # WGPU 渲染、材质、UI
│   ├── engine-physics/               # Rapier 3D 物理
│   ├── engine-app/                   # 应用入口 (winit事件循环)
│   ├── engine-platform/              # 平台抽象 (配置、输入)
│   ├── engine-scene/                 # 场景序列化 (RON)
│   └── engine-events/                # 事件系统
├── demos/                           # 游戏项目
│   ├── simple_demo/                  # WASM脚本演示
│   └── mc/                          # MC风格演示
└── config.toml                       # 全局引擎配置
```

### 1.2 执行流程

```
main.rs (engine-app)
    ↓
ToyEngineApp::new() 
    ├── 加载 project.toml
    ├── 创建 WasmRuntime (加载 game.wasm)
    └── 创建 Engine
        ↓
RunApp::run_app()
    ├── 创建 winit 窗口
    ├── App::systems() → 注册 ECS 系统
    ├── App::on_start() → 初始化渲染器
    └── 游戏循环
        ├── on_update() → 调用 WASM 更新相机
        ├── 执行 ECS 系统 (物理、渲染)
        └── 渲染帧
```

## 二、错误抽象 (Error Abstractions)

### 2.1 依赖版本不一致

**问题**: `Cargo.toml` 声明使用 `wasmtime = "22.0"`，但代码实际使用 **Wasmer** 4.x

```toml
# Cargo.toml 中声明
wasmtime = "22.0"   # 未使用

# 实际使用 (wasm_runtime.rs)
wasmer = "4"        # 实际使用
```

**影响**: 
- 依赖冗余
- 构建时可能拉取不必要的依赖

---

### 2.2 配置系统重复

**问题**: 存在多套配置文件机制

| 配置 | 位置 | 用途 |
|------|------|------|
| `config.toml` | 引擎根目录 | 全局默认项目 |
| `project.toml` | demos/simple_demo/ | 项目配置 |
| `default.toml` | demos/.../materials/ | 材质配置 |

**具体代码** (`main.rs` 手动解析):

```rust
fn GlobalConfig::load() {
    // 手动解析 TOML
    for line in contents.lines() {
        if line.starts_with("project") && line.contains('=') {
            // ...手动字符串解析
        }
    }
}
```

**建议**: 应统一使用 `serde` + `toml` 反序列化，而不是手动解析

---

### 2.3 插件系统设计不完整

**问题**: 插件 trait 定义了 `finish()` 回调，但实际使用中未实现热插拔

```rust
pub trait Plugin: Send + Sync + Debug {
    fn name(&self) -> &str;
    fn build(&self, context: &mut PluginContext);
    fn finish(&self, _context: &mut PluginContext) {}  // 很少使用
}
```

**现状**: `ToyEngineApp` 在 `reload_project()` 中手动重新构建插件，而非真正的热插拔

---

### 2.4 WASM 运行时设计问题

#### 2.4.1 紧密耦合

**问题**: WASM 运行时只能控制相机，缺乏通用设计

```rust
// main.rs on_update 中硬编码调用 WASM 获取相机位置
match runtime.call_camera_func("update", dt_seconds, 0, radius, height, speed, input_mask) {
    Ok(pos) => {
        let mut query = engine.core.world.query::<&mut Camera3D>();
        camera.position = pos;
    }
}
```

#### 2.4.2 f32 传递方式原始

```rust
/// 将 f32 转换为 i32 位表示
fn f32_to_i32_bits(f: f32) -> i32 {
    i32::from_le_bytes(f.to_le_bytes())
}
```

**问题**: 这种手动位操作容易出错，应考虑使用 Wasmer 的 typed functions

#### 2.4.3 错误处理薄弱

```rust
let _ = update_func.call(&mut self.store, &[...]);  // 忽略错误
```

**应**: 记录错误日志或返回错误

---

### 2.5 输入系统设计缺陷

**问题**: `InputCode` 跨 crate 重复定义

```rust
// engine-core/src/input/mod.rs
pub use engine_platform::input::{InputCode, ...};

pub struct InputState { ... }
```

**问题**: 输入状态在 ECS 和 Rapier 之间重复传递

```rust
// main.rs - 从 ECS 获取输入，构建 mask 传给 WASM
if let Some(ecs_input) = engine.core.world.get_resource::<InputState>() {
    for digit in 1..=4 {
        if ecs_input.is_pressed(key) {
            input_mask |= 1 << (digit - 1);
        }
    }
}
```

---

### 2.6 项目热重载实现粗糙

**问题**: `reload_project()` 手动清理所有实体

```rust
// 1. 清空 ECS 世界中的实体
let entities: Vec<Entity> = world.query::<Entity>().iter(&world).collect();
for entity in &entities {
    world.despawn(*entity);  // 暴力清理
}
```

**问题**: 可能导致资源泄漏、状态不一致

---

### 2.7 渲染系统内部耦合

**问题**: 渲染器混合了多种职责

```rust
pub mod renderer;    // 渲染器主逻辑
pub mod passes;     // Render Pass
pub mod uniforms;   // GPU Uniform
pub mod materials; // 材质系统
pub mod ui;        // UI 系统
pub mod grid;      // 网格地面
```

这些模块之间没有清晰的边界

---

## 三、待抽象点 (Missing Abstractions)

### 3.1 统一配置抽象

**现状**: 多种配置格式和加载方式

| 模块 | 配置方式 |
|------|----------|
| 全局配置 | 手动解析 TOML |
| 项目配置 | `ProjectConfig::load_or_default()` |
| 材质配置 | TOML 文件 |
| 场景配置 | RON 文件 |

**建议**: 抽象出统一配置 trait

```rust
pub trait Config: serde::de::DeserializeOwned {
    fn load(path: &Path) -> Self;
    fn reload(&mut self, path: &Path);
}
```

---

### 3.2 资源生命周期管理

**现状**: 资源手动加载，无统一生命周期

```rust
// scene loader 中手动 spawn
commands.insert(MeshRenderable { mesh_id, material_id });
// GPU 资源在 render 时按需加载
```

**缺失**:
- 资源引用计数
- 资源预加载/异步加载
- 资源卸载策略

---

### 3.3 事件/消息系统

**现状**: `engine-events` crate 存在但未充分使用

**缺失抽象**:
- 引擎内部事件 (实体创建/销毁、组件变更)
- 游戏脚本事件 (碰撞、输入)
- 跨系统通信

---

### 3.4 序列化框架

**现状**: 场景使用 RON，硬编码字段映射

```rust
// 硬编码每个字段
pub struct SceneEntity {
    pub transform: Option<TransformData>,
    pub renderable: Option<RenderableData>,
    // ... 20+ 字段
}
```

**建议**: 使用 `bevy_reflect` 实现自动序列化/反序列化

---

### 3.5 脚本系统通用抽象

**现状**: WASM 仅用于相机控制

```rust
// main.rs - 硬编码调用
runtime.call_camera_func("update", ...)
```

**缺失**:
- 通用脚本接口 trait
- 脚本生命周期 (init, update, shutdown)
- 脚本与 ECS 的交互 API

---

### 3.6 窗口/平台层抽象

**现状**: 窗口直接传给 Engine

```rust
pub struct Engine {
    pub window: Option<&'static Window>,
    pub ctx: Option<SurfaceContext<'static>>,
    pub main_renderer: Option<MainRenderer>,
}
```

**问题**: `&'static Window` 是hack，应使用引用计数或 handle

---

### 3.7 渲染管线抽象

**现状**: 硬编码渲染流程

```rust
// RenderPlugin::setup
renderer.collect_from_world(world);
// 渲染器内部
for pass in passes { pass.execute(); }
```

**缺失**:
- RenderGraph 定义 (D3D12/Vulkan 风格)
- Pass 间依赖声明
- 动态 Pass 创建/删除

---

### 3.8 音频系统未集成

**问题**: `Cargo.toml` 声明 `kira = "0.12"`，但未实际使用

```toml
# 音频
kira = "0.12"  # 未使用
```

**应**: 集成音频系统或移除依赖

---

### 3.9 物理系统与 ECS 同步方式原始

**问题**: 手动同步 Transform

```rust
pub fn sync_physics_to_transform_system(
    physics_context: Res<PhysicsContext>,
    mut transform_query: Query<(&PhysicsHandle, &mut Transform)>,
)
```

**建议**: 考虑使用 `bevy_physics` 风格的设计，或自动同步系统

---

### 3.10 日志/调试系统

**现状**: 使用 `tracing`，但缺乏分级调试能力

**缺失**:
- 运行时日志级别调整
- 性能面板 (FPS、Draw Calls、内存)
- 实体调试器

---

## 四、后续开发方向

### 4.1 短期 (1-3个月)

| 优先级 | 任务 | 原因 |
|--------|------|------|
| 高 | 统一配置系统 | 减少重复代码 |
| 高 | 完善脚本系统抽象 | 当前 WASM 只能控制相机 |
| 高 | 资源生命周期管理 | 防止内存泄漏 |
| 中 | 集成音频系统 | 依赖已声明 |
| 中 | 添加性能统计 UI | 调试必备 |

### 4.2 中期 (3-6个月)

| 优先级 | 任务 | 原因 |
|--------|------|------|
| 中 | RenderGraph 抽象 | 提升渲染管线灵活性 |

| 中 | 自动序列化 (bevy_reflect) | 简化场景/配置加载 |

## 五、总结

这是一个**架构良好的 Rust 游戏引擎原型**，具备:
- ✅ 清晰的模块划分 (7个crate)
- ✅ ECS 架构 (bevy_ecs)
- ✅ 插件系统
- ✅ WASM 脚本热更
- ✅ 物理集成 (Rapier)
- ✅ 渲染管线 (WGPU)

**但存在以下主要问题**:
1. **配置系统碎片化** - 多套配置，手动解析
2. **WASM 脚本系统不完善** - 只能控制相机，无通用接口
3. **资源管理缺失** - 无生命周期、无异步加载
4. **部分依赖未使用** - wasmtime, kira
5. **热重载实现粗糙** - 暴力清空实体

**建议后续重点**: 完善脚本系统抽象、统一配置管理、资源生命周期管理这三个核心痛点。
